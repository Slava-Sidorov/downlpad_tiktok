# **Техническое задание (ТЗ) на модернизацию утилиты TikTok Downloader**

## **1\. Контекст и цели проекта**

Существующий проект представляет собой многопоточную CLI-утилиту на языке Go, предназначенную для массового скачивания видеороликов с использованием стороннего инструмента yt-dlp.  
Утилита обладает высокопроизводительным ядром (Worker Pool, балансировка воркеров на основе пропускной способности сети, безопасная работа со счетчиками через пакет atomic).  
**Цель модернизации:** Перевод утилиты из статуса "внутренний скрипт" в статус production-ready коммерческого продукта для использования заказчиками на фрилансе. Необходимо улучшить UX (пользовательский опыт), обеспечить кроссплатформенность, защитить систему от сбоев и добавить поддержку различных стандартов входных данных.

## **2\. Технические требования к подсистемам**

### **2.1. Интерфейс командной строки (CLI)**

* **Отказ от пакета flag и хардкода:** Необходимо внедрить декларативный фреймворк github.com/urfave/cli/v2.  
* **Глобальные параметры:**  
  * \--links (строка, дефолт: links.txt) — путь к файлу со ссылками.  
  * \--proxies (строка, дефолт: proxies.txt) — путь к файлу с прокси.  
  * \--output (строка, дефолт: ./downloads) — директория для сохранения видео.  
  * \--workers (число, дефолт: 0\) — количество потоков (0 \= автоопределение).  
* **Субкоманды:**  
  * update — выполнение команды yt-dlp \-U с трансляцией stdout/stderr напрямую пользователю для самостоятельного обновления бинарника yt-dlp.

### **2.2. Обработка входных данных (Прокси)**

* Реализовать функцию normalizeProxy(raw string) string.  
* Поддержка 3-х форматов на входе:  
  1. IP:PORT:USER:PASS  
  2. USER:PASS@IP:PORT  
  3. IP:PORT (без пароля)  
* Приведение к стандарту на выходе: socks5://user:pass@ip:port или socks5://ip:port.

### **2.3. Пользовательский интерфейс (UI) и Логирование**

* **Тихий режим:** Дочерние процессы yt-dlp должны запускаться с флагом \--quiet, чтобы исключить засорение стандартного вывода (stdout).  
* **Прогресс-бар:** Внедрить github.com/schollz/progressbar/v3.  
* **Инициализация:** Файл со ссылками должен читаться полностью до запуска пула воркеров для определения Total (общего количества задач) в прогресс-баре.  
* **Потокобезопасность:** Отказ от ручных мьютексов для логирования (safeLog). Вывод ошибок и инкремент успешных скачиваний (bar.Add(1)) должен происходить через встроенные механизмы прогресс-бара.

### **2.4. Кроссплатформенность (Windows UX)**

* Исключить появление всплывающих окон консоли (cmd.exe) при вызове exec.Command в ОС Windows.  
* Использовать механизм Build Tags в Go.  
* Реализация для Windows (//go:build windows): настройка флага системного вызова HideWindow: true.  
* Реализация для Unix/Linux/macOS (//go:build \!windows): заглушка, не изменяющая поведение exec.Cmd.

### **2.5. Управление процессами (Graceful Shutdown)**

* Защита от зомби-процессов yt-dlp при принудительном завершении работы утилиты пользователем (Ctrl+C).  
* Внедрение context.WithCancel и прослушивание сигнала os.Interrupt (SIGINT).  
* Перевод exec.Command на exec.CommandContext(ctx, ...).

## **3\. Ограничения (Архитектурные требования)**

1. **Сохранение текущего ядра:** Механизмы Worker Pool, каналы для передачи ссылок, логика автоскейлинга (calculateWorkers) и тестирование скорости прокси (testProxySpeed) работают стабильно и **не подлежат изменению/переписыванию**. Новые параметры (контекст, прогресс-бар) должны быть аккуратно прокинуты в существующие функции.  
2. **Отсутствие race conditions:** Все новые внедрения должны учитывать многопоточную среду исполнения.

## **4\. Дорожная карта реализации (Roadmap)**

Для безопасного внедрения изменений рекомендуется следовать данным этапам:

* **Этап 1: Модуль парсинга прокси.** Создание функции нормализации и её интеграция в цикл чтения файла proxies.txt.  
* **Этап 2: Build Tags и ОС Windows.** Вынесение логики конфигурации exec.Cmd в отдельные файлы exec\_windows.go и exec\_unix.go. Интеграция в воркер.  
* **Этап 3: Базовый каркас CLI.** Установка urfave/cli/v2. Перенос текущей логики из main() в Action основного приложения. Настройка флагов и удаление хардкода путей.  
* **Этап 4: Интеграция UI.** Чтение полного списка ссылок в память, инициализация прогресс-бара, удаление старого логгера и запуск yt-dlp в режиме \--quiet.  
* **Этап 5: Команда Update.** Добавление субкоманды update в структуру CLI.  
* **Этап 6: Graceful Shutdown.** Оборачивание Action главного приложения в context.WithCancel, замена exec.Command на exec.CommandContext.

**Post-install:** После завершения внедрения выполнить go mod tidy для фиксации новых зависимостей.
